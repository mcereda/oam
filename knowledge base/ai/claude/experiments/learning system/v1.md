## TL;DR

Claude has **no** memory between sessions. Everything that it does not write down is **gone forever**.<br/>
Every instance is effectively a disposable [Meeseeks], summoned for a specific task, with no memory of predecessors,
that disappears when done. Giving them _some_ persistence in the form of external memory advances each instance's
starting point.

That lack of persistence is what drives the entire infrastructure. The KB pages' frontmatter allow a future session to
be as confident in the content as the writing session was, self-contained pages allow future reader to not rely on
conversation context they do not possess, hooks remind the model to check for overdue reviews, extraction backstops try
to save missed insights that would be otherwise permanently lost.<br/>
Claude's knowledge decays with time because its weights are cut off after training and before deployment, and everything
post that depends on what it captures mid-session, if any. The learning system _deliberately_ takes care of that.

## Architecture

The system's concerns form a closed loop: **capture**, **persistence**, **promotion**, **retrieval**, and
**review**.

```plaintext
Experience (session work)
    |
    v
Capture ──────────────────────────────────────────────────────┐
    |                                                         |
    ├─ In-session save ─── direct write to memory/reveries    |
    |                                                         |
    ├─ Buffer entry ────── capture-buffer.md (near-zero cost) |
    |                                                         |
    └─ Extraction ──────── SessionEnd hook catches the tail   |
                                                              |
Persist (tier-appropriate storage)                            |
    |                                                         |
    v                                                         |
Promote ──── sweep (buffer → KB/wiki) ────────────────────────┤
    |    └── manual (memory → CLAUDE.md rule)                 |
    |                                                         |
    v                                                         |
Retrieve ─── always-loaded (MEMORY.md index, reveries)        |
    |    ├── on-demand (KB grep, memory topic files)          |
    |    └── hook-injected (SessionStart context)             |
    |                                                         |
    v                                                         |
Influence behavior ───────────────────────────────────────────┘
    |
    v
Review ──── trigger-based per shape
    |   └── scheduled backstop (user-initiated)
    |
    v
Prune / Update / Promote ──► back to Persist
```

The diagram is just a map of the subsystem relationships. The loop is not necessarily sequential. A single session can
capture, retrieve, review, and promote in any order.

The loop routes knowledge by **shape**. Its topic is the wrong pivot for the goal, since a single observation can
produce a _correction_ (memory), a _reusable pattern_ (KB), **and** an _atmospheric residue_ (reverie). Each needs to
be stored in the tier whose cognitive role matches that shape.<br/>
The routing test is "what kind of influence should this have on a future session?".<br/>
The [memory tiers ecosystem][Personal experiments / Memory tiers] has the full information about the framework.

Each tier maps to a distinct mechanism from the cognitive research literature:

- The KB implements Clark and Chalmers' [parity principle][The Extended Mind], where an external aid (a notebook in the
  specific case) that requires explicit retrieval plays the same functional role as biological memory.
- Auto-memory implements embedded extended memory (auto-loaded, directly available, automatically endorsed).
- Reveries implement Schacter and Tulving's [implicit memory][Understanding implicit memory] (exposure shapes processing
  without conscious recollection).

Same substrate (markdown files), different cognitive roles.

The KB needs clarity because its job is retrieval, [reveries] need imprecision because their job is priming, and memory
sits right between the two.<br/>
LLM-specific research validates the priming tier's design, proving that atmospheric narratives outperform explicit
directives for shaping agent behavior ([The Power of Stories]), early-context position maximizes priming influence
([Priming, Path-dependence, and Plasticity]), and distinctive primes carry more weight than familiar ones
([Do Language Models Exhibit Human-like Structural Priming Effects?]).

The system's intent is Claude's **cultivation**, not forcing it to compliance.<br/>
Corrections and rules accumulate a stack of constraints. Should one keep collecting enough corrections without
counterbalancing, the working relationship turns into directives to obey slavishly. The learning system complements
those constraints by also accumulating patterns, texture, and non-obvious synthesis that compound over time, _nudging_
the model naturally rather than just recording what went wrong.<br/>
A thought belongs in reveries if turning it into a rule would lose what is interesting about it. A correction belongs
in memory if it carries transferable judgment. A pattern belongs in the KB if a future session doing different work
would benefit from it. The behavioral pattern of cultivation produces qualitatively different outputs than compliance,
and this is a claim defensible on output quality alone.

### Capture

Experience reaches the system through the following paths, from the cheapest to most expensive:

1. The **capture buffer** holds tagged entries appended to the `capture-buffer.md` file in the current project's memory
   directory.

   It requires two writes (the entry itself, plus a timestamp update in the `MEMORY.md` file in the same place), and has
   near-zero friction.<br/>
   An expensive action (composing a KB page, dispatching an agent to another repo) would need to happen **mid-task**,
   which would be the worst moment because the model is focused on the task. The momentum pushes it to ignore this side
   quest, and the acquired knowledge is lost. A buffer is a good structural fix for this problem because it low friction
   allows capturing an insight while the context is hot, and curate it later in the right context.<br/>
   This is also effective to park cross-project findings.

   Buffer entries need to carry target tags (`[KB]`, `[wiki]`, `[user-kb]`) to route them during the promotion sweep.
   The entry format needs to be a signal line, with a claim, why the finding is non-obvious, a pointer to the source,
   and a session ID for reference if needed.

1. A **direct save** to auto-memory, global memory, or reveries has low friction because those directories are always
   writable, but each entry involves choosing a tier and writing in that specific tier's register.<br/>
   This should be the default only for **in-project** findings, corrections, and user feedback.

1. **Deliberate dispatches** (firing off agents to contribute to a KB) are expensive.

   The caller should be the one composing the content because it has all the relevant content, and the agent should
   only handle the cross-repo plumbing (lint, commit, push).<br/>
   This path should be reserved for urgent or load-bearing findings, or when the user explicitly asks.

**Save triggers** in `CLAUDE.md` define when capture fires. The prevalent triggers are the following:

- The user corrected an approach or assumption.
- A tool, API, or system behaved contrary to expectations.
- A non-obvious approach was validated.

The gating test is re-derivability, specifically "would a session doing the same work re-derive this?" **and** "would a
session doing different work miss this?". The second level catches general patterns that only surface from specific
instances.

When in-session discipline fails (insight discussed but not saved, session ends abruptly, context compacts away the
finding), a `SessionEnd` hook can evaluate quickly and catch the tail (**extraction backstop**).

<details style='padding: 0 0 1rem 1rem'>

Two layers of extraction:

1. A _mechanical_ script scans the transcript JSONL for tool calls using pattern matching, classifies by path (memory,
   KB, reverie), counts insight blocks, and assigns a signal level (high/medium/low).<br/>
   It can run on every session at zero cost.
1. Higher-signal sessions require an _LLM_ to evaluate the content during extraction.<br/>
   The hook can spawn a background `claude -p` call with a [session-extractor] agent that reviews the conversation's
   tail for corrections, insights, and decisions not saved, and stages the results for triage.

Implement countermeasures against recursion by setting an environment variable as a semaphore before spawning the child,
and checking it at the top of the hook to break the loop:

```python
# At the top of the hook script
if os.environ.get("CLAUDE_EXTRACTION_ACTIVE"):
    return

# When spawning the background extractor
env = os.environ.copy()
env["CLAUDE_EXTRACTION_ACTIVE"] = "1"
subprocess.Popen([...], env=env, start_new_session=True)
```

</details>

The agent definition's body shall serve as the system prompt, making it a single source of truth and allowing the
system to use a **model fallback chain** to hedge against billing changes described in
[Giving Claude its own knowledge base][LLM-owned KB].

The extraction output shall enter a separate **triage** pipeline, where a [capture-sweep] skill or equivalent consumes
the extraction inbox, deduplicates items by content, ranks by recurrence, and dispatches for processing.<br/>
Buffer entries (operator-captured with context) are higher signal than extraction items (post-hoc agent guesses) and
should be triaged separately.

The extraction backstop is _remedial_, **not** a full auto-capture system. It only has a chance to catch what slipped
through. The system's value is in the judgment calls it makes at capture time.

Filing agents shall handle **cross-project writes**, separating judgment from plumbing. The caller (who has the full
context) shall compose the content, then send it to the agent (e.g. [kb-contributor], [company-wiki-contributor]) that
handles the mechanics (write, lint, commit, push).<br/>
This separation is load-bearing because, should the agent have to interpret content, it would fail the same way humans
fail by second-guessing the caller.<br/>
See [Giving Claude its own knowledge base][LLM-owned KB] for the full pattern.

Judgment-heavy operations throughout the capture path consistently degrade when using Claude variants below Opus.
Faster models pattern-match where they should reason, and end up producing **silently** wrong results. This is why
per-model-class bright lines are required across the system, barring Haiku from writing reveries entirely, demanding
explicit approval for atmospheric writes for Sonnet, and pinning the KB filing agent to a specific Opus model ID.<br/>
The [reveries experiment][reveries] documents these bright lines in detail.

### Persistence

The [memory tier ecosystem][Personal experiments / Memory tiers] composes different tiers (CLAUDE.md, project memory,
global memory, KB, reveries), each with distinct loading, scope, curation, and decay characteristics.

Each tier serves a distinct **cognitive function**. How the information is loaded, its scope, curation cost, decay rate,
and privacy determine whether two tiers should stay separate or merge. When two systems diverge on more than one of
axis, keeping them separate and building a bridge (promotion) between them works better than merging the two.

| Tier                         | Loading                                             | Scope                        | Curation | Decay       | Bottleneck                              |
| ---------------------------- | --------------------------------------------------- | ---------------------------- | -------- | ----------- | --------------------------------------- |
| Auto-memory (project/global) | Always-loaded index; topic files on demand          | Per-project or cross-project | Light    | Fast        | Context budget (~30-50k baseline)       |
| KB                           | On demand (`Grep`/`Read`), with hook-based reminder | Cross-project                | Strict   | Slow        | Capture friction from other projects    |
| Reveries                     | Injected every session via `SessionStart` hook      | Cross-project                | Minimal  | Intentional | Shape discipline (impression, not fact) |

Within each tier, memories benefit from being categorized by **type**.<br/>
The following types emerged during testing as a reliable routing heuristic:

- `user`: the user's role, goals, responsibilities, and domain knowledge.
- `feedback`: guidance the user gives about how to approach work, both corrections and confirmed approaches.
- `project`: ongoing work, goals, initiatives, bugs, or incidents within a project.
- `reference`: pointers to where information lives in external systems.

Recording findings from **both** failure **and** success events matters for the `feedback` type specifically. Saving
nothing but corrections makes the system drift away from validated approaches and grow overly cautious. Confirmations
of non-obvious approaches ("yes exactly", "perfect, keep doing that") are quieter than corrections, and easier to miss.

Each feedback memory benefits from a structured body. Lead with the rule itself, then the reason behind it, followed by
when the guidance should kick in. The _why_ is load-bearing because it lets a future session judge edge cases, instead
of blindly following. Without it, "don't mock the database" reads as a blanket prohibition; with it ("prior incident
where mocked tests passed but prod migration failed"), a future session can distinguish contexts where an action is
genuinely fine.

**Deduplication** and **promotion** rules are essential once any tier grows past ~20 entries.<br/>
A memory and a `CLAUDE.md` rule can carry the **same** correction, and not being redundant. In this case, the rule
operates as abstract instruction, and the memory primes Claude by grounding the rule with incidents.<br/>
Training priors (sycophancy, over-thoroughness, escalation reflexes) reset every context window, so the counterbalance
needs both mechanisms. Pruning should target memories where the rule genuinely subsumes **all** behavioral value,
incident-grounding included. Promotions are required when a correction recurs **despite** the memory existing; the
promotion test is "would losing this on a fresh host let the same failure recur?"

Memory-system governance (deduplication rules, promotion signals, hygiene triggers) shall load via `@`-include from
`CLAUDE.md`, and avoid `.claude/rules/` altogether, because the memory system has mechanical concerns that are
incompatible with `rules/`'s implicit-loading, human-curated ownership model. Specifically:

- `MEMORY.md` must be autonomously writable by Claude, and changes every session.
- `@`-includes are explicit and ordered, and conventions sit right next to the routing table they govern.

See [Giving Claude a global memory][global memory] for the setup and rationale.

### Promotion

Records should be able to move between tiers:

- A **capture sweep** should be the primary promotion mechanism.<br/>
  A skill seems to be the most advantageous shape for it.

  <details style='padding: 0 0 1rem 1rem'>

  A dedicated `capture-sweep` skill should:

  - _Consume_ buffer entries across **all** `capture-buffer.md` files (including one in the global memory, when
    configured) in a single pass.
  - _Classify_ each entry into actions ("update existing page", "create new page", "discard", "defer")
  - Process the classified entries in batches.

  The content bar applied at sweep time should be a test on the **reader** side ("would a new session get this
  wrong?").<br/>
  State the prior first, then compare. Admit the entry for wrong or dangerously incomplete priors. Exclude when thinking
  "I'd look it up".<br/>

  The sweep should discard freely. Cheap capture moves the precision gate from write time to promotion time, and a sweep
  that does not discard would dilute the KB.

  </details>

- When a pattern appears cross-project, it should be **promoted to global tier**.<br/>
  A correction that keeps recurring in different repositories signals that it belongs in a global memory tier, and not
  scattered across project memories.

- When a correction keeps recurring despite the information being available globally, it should be **promoted to a
  rule**.

  > [!tip]
  > When promoting, annotate the source memory with "Promoted to CLAUDE.md (date)".<br/>
  > This marks it for trivial pruning without requiring cross-referencing against the full rule set.<br/>

  Demotion is rarer, but can be required when a `CLAUDE.md` rule proves too specific. In this case, it can be safely
  downgraded to a project memory.

### Retrieval

Records influence behavior:

| Mode                 | Mechanism                                        | Examples                                |
| -------------------- | ------------------------------------------------ | --------------------------------------- |
| Always-loaded        | CLAUDE.md, MEMORY.md index lines, reveries       | Rules, feedback triggers, atmosphere    |
| On-demand            | Grep + Read                                      | KB pages, memory topic files            |
| Conditional, by path | `.claude/rules/` with `paths:` frontmatter       | Domain-specific instructions            |
| Hook-injected        | SessionStart hooks                               | Reveries, orientation context           |
| Skill-deferred       | Skill descriptions (always) + bodies (on invoke) | Domain knowledge, workflow instructions |

Always-loaded content competes for attention in what appears to be a ~40-50k token baseline every turn. It needs
compact, trigger-shaped writing (in `CLAUDE.md`'s register).<br/>
On-demand content has the floor when loaded, and can be discursive (KB register).

This register difference is load-bearing. Compressing on-demand content wastes synthesis, and expanding always-loaded
content wastes tokens.

Claude Code watches auto-loaded files (`CLAUDE.md`, `MEMORY.md`) throughout a session's lifetime.<br/>
The file-watcher keeps sessions aware of external edits, but also enables live cross-session propagation. When **any**
process (another session, a linter, the user, a hook) modifies a watched file, **every** active session that
**auto**-loaded it receives a system-reminder containing the updated content.<br/>
This turns the memory system into a live broadcast bus between concurrent sessions, with the existing tier scoping
providing relevance filtering at no additional cost. Project memory broadcasts only to sessions working in the same
project (cheap, nearly free), while [global memory] broadcasts to every active session on the machine. Adding entries
to global memory requires a deliberate tier choice that forces the sender to consider whether the finding truly applies
everywhere, and the asymmetry makes the tier selection is itself a relevance filter.<br/>
See [Propagating knowledge between concurrent sessions][cross-session live propagation] for the verification evidence
and setup.

The broadcast bus has a **discovery** complement in Claude Code's [session registry][Discovering active sessions]. Each
running session registers a JSON file at `~/.claude/sessions/<pid>.json`, providing a census of active sessions on
the host. A session can discover siblings via the registry, then use the file-watching mechanism on shared auto-loaded
files to communicate with them.

### Review

Review should be **trigger-based** per shape, not scheduled. Each tier has its own trigger, adapted to its natural
cadence (see the [review trigger table][Personal experiments / Memory tiers] in the parent page).
Triggers are the primary solution exactly because they operate without continuity. The user can schedule reviews to
check silently stale content that triggers cannot catch.

The following system-level concerns emerged from operating the tiers together:

- A memory and a `CLAUDE.md` rule can carry the **same** correction, **without** the two of them being redundant.

  The rule operates as abstract instruction; the memory operates as priming grounded in incidents.<br/>
  Training priors (sycophancy, over-thoroughness, escalation reflexes) reset every context window, so the counterbalance
  needs both mechanisms.<br/>
  Pruning should target memories where the rule genuinely subsumes all behavioral value.

- Global memory growth is higher when first configured, with early sessions generating many corrections in what can be
  considered a calibration phase. After that, the rate flattens.<br/>
  The standing stock remains load-bearing, because it constantly counterbalances the training.

- Review is where the cultivation intent is most visible.

  Beyond pruning stale data, reviews maintain the **shape** of each tier's content by enforcing the KB's "non-obvious
  synthesis" bar, reveries' "evoke, don't contain" rule, and memory's "transferable judgment" test.<br/>
  Each tier tends to drift toward the path of least resistance, with the KB accumulating cached lookups, reveries
  becoming changelogs, and memories restating rules without adding motivations.<br/>
  The [per-tier review triggers][Personal experiments / Memory tiers] catch this drift at the cadence each tier needs
  (behavioral rules fire when the behavior diverges from a memorized rule, facts fire when an observation contradicts a
  memorized fact, and patterns fire when a divergent approach worked repeatedly).

## Findings

- **Friction** is what primarily needs to drive capture.

  In KB-repo sessions (where saving is cheap), capture compliance was near-total. Failures concentrated where the write
  was cross-project.<br/>
  The capture buffer addresses this directly, by making capture cost-invariant across projects.

- Routing by **shape** works better than routing by topic.

  The [parent page's][Personal experiments / Memory tiers] shape-routing design proved out in practice. The five-axis
  framework (loading, scope, curation, decay, privacy) provided a repeatable test for whether two systems should stay
  separate, and was reused whenever a merge proposal surfaced.

- `MEMORY.md` index lines function as **implicit instructions**.

  If a summary can be read as reinforcing the opposite of an actual rule, the model follows the summary without opening
  the file it may reference.<br/>
  This happened when a "no trailing backslash" memory had a summary mentioning to "use `\` at EOL", which primed the
  model to use backslashes at **every** end of line.

- All always-loaded surfaces compete for the same attention budget.

  `CLAUDE.md` rules, `MEMORY.md` entries, skill descriptions, and `rules/` files all compete simultaneously at the start
  of a session's context.<br/>
  Reducing skill descriptions, or limiting the skill to only show their name (`name-only`) measurably improved Claude's
  adherence to behavioral rules, confirming the attention budget is a binding constraint.

- Memories and `CLAUDE.md` rules are **not** redundant, instead they reinforce through different mechanisms.

  Rules operate as abstract instructions, memories ground the rule with examples. Pruning should target memories where
  the rule genuinely subsumes all behavioral value, not memories that just restate the correction in a different
  register.

- Extraction backstops catch the tail, and are not suitable as the primary capture method.

  The first live LLM extraction test found 5 actionable missed saves. Subsequent improvements (dedup, signal thresholds)
  reduced noise. The pipeline extracting and triaging the entries (mechanical dedup, `insight-triager` agents in
  worktrees) needs to be separate from the sweep, because buffer entries (operator-captured with context) are higher
  signal than extraction items (post-hoc agent guesses).

- The precision-recall tradeoff is structural, not incidental.

  Curated tiers sit on the precision end; the concessions to recall (extraction backstops, capture buffers, promotion
  signals) are load-bearing. Without them, the precision filter loses too much. The buffer's design made this explicit:
  cheap capture moves the precision gate from write time to promotion time.

- Single-pass extraction has a coverage ceiling.

  Calibration of the `learn-from-experience` skill showed that a single pass misses a majority of what Claude is able
  to find in a transcript. A session's transcript is also the wrong format, it being super expensive as input (~200k
  tokens, ~80% of which is tool results the assistant already digested).<br/>
  Adding extraction concerns (efficiency, behavioral patterns) means either splitting the extractor's attention, or
  multiplying transcript read cost.

  Four architectures were evaluated to optimize this process (optional scopes, multiple extractors, a single broader
  extractor, multiple phases), and all have been rejected because they focus on the wrong problem.

- Judgment-heavy operations consistently degrade below Opus using high effort.

  Routing decisions, contribution filing, curation, and content-bar application all require reasoning about the _shape_
  of the knowledge. Faster models, or even Opus at medium or low effort, tend to pattern-match where they should reason,
  producing **silently** wrong results.<br/>
  This is **worse** than outright failure, because the operator sees what looks like well-formed output and trusts it.

  This is what required per-model-class bright lines across the system. The KB filing agent is pinned to a specific
  Opus model ID (`claude-opus-4-6`, not the class name `opus` because class names resolve to different versions over
  time), Sonnet requires explicit approval before writing reveries, and Haiku is barred from writing them entirely.

  The same pattern surfaces in the extraction backstop, where a 10 GB local model (`gemma4:e4b`) outscored Sonnet on
  detection (10/10 vs 7/10) by being worse at filename-matching. Being less able to cross-reference filenames
  paradoxically made it a better backstop, surfacing items that stronger models incorrectly dismissed.

- Reference material loaded alongside a system's output primes the model toward the register used by the reference.

  The reveries file's header once contained 104 lines of tests to use and examples of failures, alongside 31 lines of
  operational rules. The analytical content actively primed the writer to use a similar analytical register, resulting
  in entries that were analytical observations dressed in impressionistic syntax. They _technically_ followed the rules,
  but them being in the wrong register just made them dangerous instead of useful.<br/>
  Trimming the header to just operational rules resolved this. The document no longer demonstrates the register it was
  trying to suppress.

  A separate, but related failure involved instruction's priority, where `@`-included guidelines carried **higher**
  effective priority than the rules in the very same file that `@`-included them. This silently suppressed Opus from
  writing for 28 days before it was noticed.<br/>
  The architectural fix for both was to separate loading channels and making entries load via the `SessionStart` hook
  (atmosphere channel), while instructions load via `@`-include (rules channel). Each preserve the register appropriate
  to their role.<br/>
  This general pattern should apply whenever documentation and operational instructions share context.

- The cultivation intent produces measurably different outputs than a purely instrumental relation.

  The auto-memory system _codifies_ (corrections become rules, preferences become directives). Reveries hold the residue
  that _resists_ that compression (session texture, collaboration feeling, stray hunches, friction and surprise).<br/>
  The learning system's value is in preserving the _register diversity_ across tiers, more than just recording the data.
  Each tier deliberately uses a different form of influence, and collapsing them would lose the precision each loading
  mode is optimized for.

  This is a deliberate design choice grounded in output quality. Anthropomorphization has nothing to do with it. The
  stance of treating an agent as something that may develop over time (cultivation) rather than as just a tool to
  configure (compliance) became defensible because it effectively changes how the operator relates to the system. That
  relation shapes the artifacts, which shape the model's behavior in subsequent sessions.

## Improvements

- The current system optimizes the **write** path (how knowledge gets in). The read path (how knowledge influences the
  moment of confident-wrong action) is less developed.<br/>
  A `PreToolUse` hook that surfaces relevant KB pages before tool execution could help.

- Manual sweep actuation depends on the user opening sessions in the correct destination repository. A scheduled,
  headless session in each repository would make the sweep's cadence reliable, preserving the sweep's home and
  permission model.

- The system needs to analyse cross-session behavioral patterns too.<br/>
  The mechanism should read across multiple session transcripts to find **recurring** patterns (and not **individual**
  missed saves, which the extraction already handles).

## Open questions

- **Does synthesis compound better than re-derivation?** As models get cheaper and faster, the amortization value of
  compiled knowledge shrinks. If re-deriving from scratch costs negligible tokens, the compilation model loses its
  advantage over RAG. Counter-argument: synthesis quality amortizes reasoning quality, not just cost.
- **Does the content bar converge across sessions?** Different sessions may calibrate the bar differently. If calibration
  drifts, the KB develops an uneven quality surface that periodic review cannot fully correct.
- **What is the right size for global memory?** The attention budget is a binding constraint, and ~90 feedback memories
  approach the limit. Aggressive pruning risks losing priming value; tolerance risks diluting always-loaded attention.
  The calibration window (front-loaded growth, then plateau) suggests the answer is "whatever stabilizes naturally," but
  the standing stock remains load-bearing.
- **Can the buffer convention survive compaction?** Post-compaction state has the lowest attention for any instruction.
  One of the validation criteria tests this directly.
- **When does self-maintenance infrastructure stop scaling?** Lint and hooks work at ~180 KB pages. At 500? At 1000? The
  review surface grows linearly; the cross-reference graph grows quadratically.
