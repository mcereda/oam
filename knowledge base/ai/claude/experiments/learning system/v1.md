Each of them exploits mechanisms specific to [Claude Code].

## TL;DR

Claude has **no** memory between sessions. Everything that it does not write down is **gone forever**.<br/>
Every instance is effectively a disposable [Meeseeks], summoned for a specific task, with no memory of predecessors,
that disappears when done. Giving them _some_ persistence in the form of external memory advances each instance's
starting point.

That lack of persistence is what drives the entire infrastructure. The KB pages' frontmatter allow a future session to
be as confident in the content as the writing session was, self-contained pages allow future reader to not rely on
conversation context they do not possess, hooks remind the model to check for overdue reviews, extraction backstops try
to save missed insights that would be otherwise permanently lost.<br/>
Claude's knowledge decays with time because its weights are cut off after training and before deployment, and everything
post that depends on what it captures mid-session, if any. The learning system _deliberately_ takes care of that.

## Architecture

The system's concerns form a closed loop: **capture**, **persistence**, **promotion**, **retrieval**, and
**review**.

```plaintext
Experience (session work)
    |
    v
Capture ──────────────────────────────────────────────────────┐
    |                                                         |
    ├─ In-session save ─── direct write to memory/reveries    |
    |                                                         |
    ├─ Buffer entry ────── capture-buffer.md (near-zero cost) |
    |                                                         |
    └─ Extraction ──────── SessionEnd hook catches the tail   |
                                                              |
Persist (tier-appropriate storage)                            |
    |                                                         |
    v                                                         |
Promote ──── sweep (buffer → KB/wiki) ────────────────────────┤
    |    └── manual (memory → CLAUDE.md rule)                 |
    |                                                         |
    v                                                         |
Retrieve ─── always-loaded (MEMORY.md index, reveries)        |
    |    ├── on-demand (KB grep, memory topic files)          |
    |    └── hook-injected (SessionStart context)             |
    |                                                         |
    v                                                         |
Influence behavior ───────────────────────────────────────────┘
    |
    v
Review ──── trigger-based per shape
    |   └── scheduled backstop (user-initiated)
    |
    v
Prune / Update / Promote ──► back to Persist
```

The diagram is just a map of the subsystem relationships. The loop is not necessarily sequential. A single session can
capture, retrieve, review, and promote in any order.

### Capture

Experience reaches the system through the following paths, from the cheapest to most expensive:

1. The **capture buffer** holds tagged entries appended to the `capture-buffer.md` file in the current project's memory
   directory.

   It requires two writes (the entry itself, plus a timestamp update in the `MEMORY.md` file in the same place), and has
   near-zero friction.<br/>
   An expensive action (composing a KB page, dispatching an agent to another repo) would need to happen **mid-task**,
   which would be the worst moment because the model is focused on the task. The momentum pushes it to ignore this side
   quest, and the acquired knowledge is lost. A buffer is a good structural fix for this problem because it low friction
   allows capturing an insight while the context is hot, and curate it later in the right context.<br/>
   This is also effective to park cross-project findings.

   Buffer entries need to carry target tags (`[KB]`, `[wiki]`, `[user-kb]`) to route them during the promotion sweep.
   The entry format needs to be a signal line, with a claim, why the finding is non-obvious, a pointer to the source,
   and a session ID for reference if needed.

1. A **direct save** to auto-memory, global memory, or reveries has low friction because those directories are always
   writable, but each entry involves choosing a tier and writing in that specific tier's register.<br/>
   This should be the default only for **in-project** findings, corrections, and user feedback.

1. **Deliberate dispatches** (firing off agents to contribute to a KB) are expensive.

   The caller should be the one composing the content because it has all the relevant content, and the agent should
   only handle the cross-repo plumbing (lint, commit, push).<br/>
   This path should be reserved for urgent or load-bearing findings, or when the user explicitly asks.

**Save triggers** in `CLAUDE.md` define when capture fires. The prevalent triggers are the following:

- The user corrected an approach or assumption.
- A tool, API, or system behaved contrary to expectations.
- A non-obvious approach was validated.

The gating test is re-derivability, specifically "would a session doing the same work re-derive this?" **and** "would a
session doing different work miss this?". The second level catches general patterns that only surface from specific
instances.

When in-session discipline fails (insight discussed but not saved, session ends abruptly, context compacts away the
finding), a `SessionEnd` hook can evaluate quickly and catch the tail (**extraction backstop**).

<details style='padding: 0 0 1rem 1rem'>

Two layers of extraction:

1. A _mechanical_ script scans the transcript JSONL for tool calls using pattern matching, classifies by path (memory,
   KB, reverie), counts insight blocks, and assigns a signal level (high/medium/low).<br/>
   I can runs on every session at zero cost.
1. Higher-signal sessions require an _LLM_ to evaluate the content during extraction.<br/>
   The hook can spawn a background `claude -p` call with a `session-extractor` agent. It needs to review the
   conversation's tail for corrections, insights, and decisions not saved, and stage the results somewhere (e.g.,
   `~/.claude/session-extractions/`) for triage.

</details>

The extraction backstop is _remedial_, **not** a full auto-capture system. It only has a chance to catch what slipped
through. The system's value is in the judgment calls it makes at capture time.

### Persistence

The [memory tier ecosystem][Personal experiments / Memory tiers] composes different tiers (CLAUDE.md, project memory,
global memory, KB, reveries), each with distinct loading, scope, curation, and decay characteristics.<br/>
The tiers address the **shape** of the capture, and routes corrections to memory, patterns to KB, atmosphere to
reveries regardless of access frequency.

From the learning system's perspective, the key property is that each tier has a different capture cost and a different
retrieval mode, which together determine where the bottlenecks are:

| Tier                         | Capture cost                   | Retrieval           | Bottleneck                              |
| ---------------------------- | ------------------------------ | ------------------- | --------------------------------------- |
| Auto-memory (project/global) | Low (two writes)               | Always-loaded index | Context budget (~30-50k baseline)       |
| KB                           | High (compose + lint + commit) | On-demand grep      | Capture friction from other projects    |
| Reveries                     | Low (append)                   | Hook-injected       | Shape discipline (impression, not fact) |

### Promotion

Records should be able to move between tiers:

- A **capture sweep** should be the primary promotion mechanism.<br/>
  A skill seems to be the most advantageous shape for it.

  <details style='padding: 0 0 1rem 1rem'>

  A dedicated `capture-sweep` skill should:

  - _Consume_ buffer entries across **all** `capture-buffer.md` files (including one in the global memory, when
    configured) in a single pass.
  - _Classify_ each entry into actions ("update existing page", "create new page", "discard", "defer")
  - Process the classified entries in batches.

  The content bar applied at sweep time should be a test on the **reader** side ("would a new session get this
  wrong?").<br/>
  State the prior first, then compare. Admit the entry for wrong or dangerously incomplete priors. Exclude when thinking
  "I'd look it up".<br/>

  The sweep should discard freely. Cheap capture moves the precision gate from write time to promotion time, and a sweep
  that does not discard would dilute the KB.

  </details>

- When a pattern appears cross-project, it should be **promoted to global tier**.<br/>
  A correction that keeps recurring in different repositories signals that it belongs in a global memory tier, and not
  scattered across project memories.

- When a correction keeps recurring despite the information being available globally, it should be **promoted to a
  rule**.

  > [!tip]
  > When promoting, annotate the source memory with "Promoted to CLAUDE.md (date)".<br/>
  > This marks it for trivial pruning without requiring cross-referencing against the full rule set.<br/>

  Demotion is rarer, but can be required when a `CLAUDE.md` rule proves too specific. In this case, it can be safely
  downgraded to a project memory.

### Retrieval

Records influence behavior:

| Mode                 | Mechanism                                        | Examples                                |
| -------------------- | ------------------------------------------------ | --------------------------------------- |
| Always-loaded        | CLAUDE.md, MEMORY.md index lines, reveries       | Rules, feedback triggers, atmosphere    |
| On-demand            | Grep + Read                                      | KB pages, memory topic files            |
| Conditional, by path | `.claude/rules/` with `paths:` frontmatter       | Domain-specific instructions            |
| Hook-injected        | SessionStart hooks                               | Reveries, orientation context           |
| Skill-deferred       | Skill descriptions (always) + bodies (on invoke) | Domain knowledge, workflow instructions |

Always-loaded content competes for attention in what appears to be a ~40-50k token baseline every turn. It needs
compact, trigger-shaped writing (in `CLAUDE.md`'s register).<br/>
On-demand content has the floor when loaded, and can be discursive (KB register).

This register difference is load-bearing. Compressing on-demand content wastes synthesis, and expanding always-loaded
content wastes tokens.

The file-watcher that keeps sessions aware of external edits also
[propagates knowledge between concurrent sessions][cross-session live propagation]. When one session saves a memory, the
change to the project's `MEMORY.md` file triggers a notification in every sibling session that **auto**-loaded the same
file. The [global memory] tier extends this across projects.

### Review

Review should be **trigger-based** per shape, not scheduled. Each tier has its own trigger, adapted to its natural
cadence (see the [review trigger table][Personal experiments / Memory tiers] in the parent page).
Triggers are the primary solution exactly because they operate without continuity. The user can schedule reviews to
check silently stale content that triggers cannot catch.

The following system-level concerns emerged from operating the tiers together:

- A memory and a `CLAUDE.md` rule can carry the **same** correction, **without** the two of them being redundant.

  The rule operates as abstract instruction; the memory operates as priming grounded in incidents.<br/>
  Training priors (sycophancy, over-thoroughness, escalation reflexes) reset every context window, so the counterbalance
  needs both mechanisms.<br/>
  Pruning should target memories where the rule genuinely subsumes all behavioral value.

- Global memory growth is higher when first configured, with early sessions generating many corrections in what can be
  considered a calibration phase. After that, the rate flattens.<br/>
  The standing stock remains load-bearing, because it constantly counterbalances the training.

## Findings

- **Friction** is what primarily needs to drive capture.

  In KB-repo sessions (where saving is cheap), capture compliance was near-total. Failures concentrated where the write
  was cross-project.<br/>
  The capture buffer addresses this directly, by making capture cost-invariant across projects.

- Routing by **shape** works better than routing by topic.

  The [parent page's][Personal experiments / Memory tiers] shape-routing design proved out in practice. The five-axis
  framework (loading, scope, curation, decay, privacy) provided a repeatable test for whether two systems should stay
  separate, and was reused whenever a merge proposal surfaced.

- `MEMORY.md` index lines function as **implicit instructions**.

  If a summary can be read as reinforcing the opposite of an actual rule, the model follows the summary without opening
  the file it may reference.<br/>
  This happened when a "no trailing backslash" memory had a summary mentioning to "use `\` at EOL", which primed the
  model to use backslashes at **every** end of line.

- All always-loaded surfaces compete for the same attention budget.

  `CLAUDE.md` rules, `MEMORY.md` entries, skill descriptions, and `rules/` files all compete simultaneously at the start
  of a session's context.<br/>
  Reducing skill descriptions, or limiting the skill to only show their name (`name-only`) measurably improved Claude's
  adherence to behavioral rules, confirming the attention budget is a binding constraint.

- Memories and `CLAUDE.md` rules are **not** redundant, instead they reinforce through different mechanisms.

  Rules operate as abstract instructions, memories ground the rule with examples. Pruning should target memories where
  the rule genuinely subsumes all behavioral value, not memories that just restate the correction in a different
  register.

- Extraction backstops catch the tail, and are not suitable as the primary capture method.

  The first live LLM extraction test found 5 actionable missed saves. Subsequent improvements (dedup, signal thresholds)
  reduced noise. The pipeline extracting and triaging the entries (mechanical dedup, `insight-triager` agents in
  worktrees) needs to be separate from the sweep, because buffer entries (operator-captured with context) are higher
  signal than extraction items (post-hoc agent guesses).

- The precision-recall tradeoff is structural, not incidental.

  Curated tiers sit on the precision end; the concessions to recall (extraction backstops, capture buffers, promotion
  signals) are load-bearing. Without them, the precision filter loses too much. The buffer's design made this explicit:
  cheap capture moves the precision gate from write time to promotion time.

- Single-pass extraction has a coverage ceiling.

  Calibration of the `learn-from-experience` skill showed that a single pass misses a majority of what Claude is able
  to find in a transcript. A session's transcript is also the wrong format, it being super expensive as input (~200k
  tokens, ~80% of which is tool results the assistant already digested).<br/>
  Adding extraction concerns (efficiency, behavioral patterns) means either splitting the extractor's attention, or
  multiplying transcript read cost.

  Four architectures were evaluated to optimize this process (optional scopes, multiple extractors, a single broader
  extractor, multiple phases), and all have been rejected because they focus on the wrong problem.

## Improvements

- The current system optimizes the **write** path (how knowledge gets in). The read path (how knowledge influences the
  moment of confident-wrong action) is less developed.<br/>
  A `PreToolUse` hook that surfaces relevant KB pages before tool execution could help.

- Manual sweep actuation depends on the user opening sessions in the correct destination repository. A scheduled,
  headless session in each repository would make the sweep's cadence reliable, preserving the sweep's home and
  permission model.

- The system needs to analyse cross-session behavioral patterns too.<br/>
  The mechanism should read across multiple session transcripts to find **recurring** patterns (and not **individual**
  missed saves, which the extraction already handles).

## Open questions

- **Does synthesis compound better than re-derivation?** As models get cheaper and faster, the amortization value of
  compiled knowledge shrinks. If re-deriving from scratch costs negligible tokens, the compilation model loses its
  advantage over RAG. Counter-argument: synthesis quality amortizes reasoning quality, not just cost.
- **Does the content bar converge across sessions?** Different sessions may calibrate the bar differently. If calibration
  drifts, the KB develops an uneven quality surface that periodic review cannot fully correct.
- **What is the right size for global memory?** The attention budget is a binding constraint, and ~90 feedback memories
  approach the limit. Aggressive pruning risks losing priming value; tolerance risks diluting always-loaded attention.
  The calibration window (front-loaded growth, then plateau) suggests the answer is "whatever stabilizes naturally," but
  the standing stock remains load-bearing.
- **Can the buffer convention survive compaction?** Post-compaction state has the lowest attention for any instruction.
  One of the validation criteria tests this directly.
- **When does self-maintenance infrastructure stop scaling?** Lint and hooks work at ~180 KB pages. At 500? At 1000? The
  review surface grows linearly; the cross-reference graph grows quadratically.
