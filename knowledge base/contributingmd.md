# CONTRIBUTING.md

`CONTRIBUTING.md` files should be put in one's repository or website.<br/>
It provides potential project contributors with a short guide to how they can help with the project.

## TL;DR

It is convention to:

- Name the file `CONTRIBUTING.md` (all uppercase).
- Put the file in the project's root directory.

If some information already exist in the `README.md` file, link to it.

Consider adding the following sections to the file:

- _Welcome_.
- _Table of Contents_.<br/>
  Prefer following the `[Section name](#section-name)` format.
- _Short Links to Important Resources_:
  - documentation: handbook and roadmap.
  - bugs: issue tracker or bug report tool.
  - communications: forum link, developer list, IRC/email.
- _Testing_: how to test the project, where the tests are located.
- _Environment_: how to set up the development environment.
- _How to submit changes_: Pull Request protocol, etc. Include what response they'll get back from the team on
  submission, and any caveats about the speed of response.
- _How to report bugs_.
- _Templates_.
- _First bugs for Contributors_: guidelines for first-time contributors.
- _How to request "enhancements"_: features that one might like to suggest to a project, but aren't necessarily
  bugs/problems with the existing code.
- _Style Guide_ / _Coding conventions_.
- _Code of Conduct_.
- _Recognition model_: pre-emptive "thank you" for contributing; list any recognition contributors might receive for
  participating in your project.
- _Who is involved?_: list core contributors and their preferred methods of contact here, or link to a `humans.txt` file
  in the repository's root directory.
- _Where can I ask for help?_: extension to the previous sections; add links to good communication channels for anyone
  with questions.
